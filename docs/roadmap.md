# Trade Engine - Дорожная карта разработки

## Обзор

Эта дорожная карта описывает план разработки от текущего состояния до production-ready системы с веб-интерфейсом. Каждая фаза строится на предыдущей. Фазы последовательны, но задачи внутри фазы могут выполняться параллельно.

**Текущее состояние:** BCS REST-адаптер, реализующий интерфейс `Broker`, захардкоженная логика ребалансировки в `cmd/main.go`, нет конфигурации, нет тестов, нет оркестратора.

---

## Фаза 1: Фундамент

**Цель:** Заменить захардкоженный `main.go` конфигурируемой, оркестрируемой системой. Заложить основную архитектуру, на которой строится всё остальное.

**Оценка трудозатрат:** 3-4 недели

### 1.1 Структурированное логирование

- [ ] Настроить `log/slog` с JSON-выводом для production и текстовым выводом для разработки.
- [ ] Заменить все вызовы `fmt.Println` и `log.Fatal` на структурированный логгер.
- [ ] Добавить контекстные поля: `bot_id`, `broker`, `account_id`.

### 1.2 Система конфигурации

- [ ] Определить структуры `Config` (см. tech-spec раздел 3.1).
- [ ] Реализовать парсер YAML с `gopkg.in/yaml.v3`.
- [ ] Реализовать интерполяцию `${ENV_VAR}` для секретов.
- [ ] Добавить валидацию конфига (обязательные поля, допустимые типы расписаний, ссылки на брокеров).
- [ ] Написать `config.example.yaml` с документированными полями.

### 1.3 Интерфейс стратегии и типы

- [ ] Определить интерфейс `Runtime`: `Start`, `OnTick`, `OnSchedule`, `Stop`.
- [ ] Определить типы `State`, `Response`, `Intent`, `IntentAction`.
- [ ] Определить спецификацию протокола JSON-lines для коммуникации через subprocess.

### 1.4 Subprocess Runtime

- [ ] Реализовать `SubprocessRuntime`: запуск процесса, коммуникация через stdin/stdout JSON-lines.
- [ ] Обработка жизненного цикла процесса: запуск, проверка здоровья, graceful-остановка, принудительное завершение по таймауту.
- [ ] Реализовать обработку ошибок: крэш стратегии, невалидный JSON-ответ, таймаут stdout.
- [ ] Написать пример стратегии ребалансировки на Python (`strategies/examples/rebalance.py`).

### 1.5 Broker Proxy

- [ ] Реализовать `BrokerProxy` с валидацией intent'ов.
- [ ] Добавить rate-limiting (скользящее окно, настраиваемый per bot).
- [ ] Добавить проверку максимального размера ордера.
- [ ] Добавить whitelist разрешённых тикеров (опционально).
- [ ] Логировать каждое исполнение intent'а с ID бота для аудита.

### 1.6 Бот

- [ ] Реализовать структуру `Bot`: связывает runtime + proxy + config.
- [ ] Реализовать планировщик `runCron` с использованием `robfig/cron/v3`.
- [ ] Реализовать graceful shutdown через отмену `context.Context`.

### 1.7 Оркестратор

- [ ] Реализовать `Orchestrator`: управление жизненным циклом ботов (старт, стоп, статус).
- [ ] Реализовать `Broker Registry`: маппинг имён брокеров из конфига на реализации `Broker`.
- [ ] Связать `cmd/trade-engine/main.go`: загрузка конфига -> инициализация брокеров -> запуск оркестратора.
- [ ] Реализовать graceful shutdown по SIGINT/SIGTERM с таймаутом.

### 1.8 Миграция логики ребалансировки

- [ ] Переписать логику `internal/rebalance/` как пример стратегии на Python.
- [ ] Проверить, что она генерирует те же торговые intent'ы, что и старая захардкоженная логика.
- [ ] Удалить старый пакет `internal/rebalance/`.

### 1.9 Тесты

- [ ] Unit-тесты для парсинга и валидации конфига.
- [ ] Unit-тесты для `BrokerProxy` (валидация intent'ов, rate-limiting).
- [ ] Unit-тесты для subprocess runtime (мок stdin/stdout).
- [ ] Интеграционный тест: оркестратор запускает бота с мок-брокером и subprocess-стратегией.
- [ ] Создать `broker.Mock`, реализующий интерфейс `Broker` для тестирования.

**Результат Фазы 1:** Бинарник `trade-engine`, который читает `config.yaml`, запускает ботов с subprocess-стратегиями, исполняет сделки через БКС via `BrokerProxy` и корректно завершается.

## Фаза 3: Wasm Runtime

**Цель:** Добавить высокопроизводительный, изолированный Wasm runtime для стратегий на Go/Rust.

**Оценка трудозатрат:** 2-3 недели

### 3.1 Ядро Wasm Runtime

- [ ] Реализовать `WasmRuntime` с использованием `wazero`.
- [ ] Определить host-функции, экспортируемые в Wasm: `get_state`, `emit_intent`, `log_message`.
- [ ] Определить guest-функции, ожидаемые от Wasm: `on_init`, `on_tick`, `on_schedule`.
- [ ] Обмен данными через shared linear memory (JSON-сериализация).

### 3.2 Wasm Strategy SDK (Go)

- [ ] Создать пакет `strategies/sdk/go/` с вспомогательными типами и функциями.
- [ ] Руководство по компиляции через TinyGo и Makefile.
- [ ] Пример: моментум-стратегия на Go, скомпилированная в `.wasm`.

### 3.3 Wasm Strategy SDK (Rust) (опционально)

- [ ] Создать крейт `strategies/sdk/rust/` с вспомогательными типами.
- [ ] Цель сборки Cargo: `wasm32-wasi`.
- [ ] Пример: простая стратегия на Rust, скомпилированная в `.wasm`.

### 3.4 Пулинг Wasm-инстансов

- [ ] Компиляция `.wasm` один раз, кэширование скомпилированного модуля.
- [ ] Для cron-ботов: инстанцирование по требованию, уничтожение после выполнения.
- [ ] Для active-ботов: долгоживущие инстансы.
- [ ] Настраиваемые лимиты размера пула.

### 3.5 Тесты

- [ ] Unit-тесты для Wasm runtime (компиляция + выполнение тестового `.wasm`-модуля).
- [ ] Интеграционный тест: запуск Wasm-стратегии через полный пайплайн бота.
- [ ] Бенчмарк: сравнение задержки Wasm vs subprocess на тик.

**Результат Фазы 3:** Wasm runtime наряду с subprocess runtime. Пользователи могут писать стратегии на Go (TinyGo) или Rust и компилировать в `.wasm`.

## Фаза 5: Hot-Reload конфига и Т-Инвестиции

**Цель:** Динамическое обновление конфига без перезапуска и второй адаптер брокера.

**Оценка трудозатрат:** 2-3 недели

### 5.1 Hot-Reload конфига

- [ ] Реализовать `fsnotify`-наблюдатель за `config.yaml` и файлами стратегий.
- [ ] Реализовать diff конфигов: обнаружение добавленных, удалённых и изменённых ботов.
- [ ] `Orchestrator.Reload()`: graceful-остановка изменённых ботов, запуск новых, удаление удалённых.
- [ ] Неизменённые боты продолжают работать без прерывания.
- [ ] Обработка пограничных случаев: невалидный новый конфиг (продолжать работу со старым), права доступа к файлам.

### 5.2 Hot-Reload файлов стратегий

- [ ] Отслеживание путей к файлам стратегий из конфига.
- [ ] При изменении файла стратегии: остановка бота, перезагрузка стратегии, перезапуск бота.
- [ ] Для Wasm: рекомпиляция модуля из нового `.wasm`-файла.
- [ ] Для subprocess: перезапуск процесса с новым скриптом.

### 5.3 Адаптер Т-Инвестиций

- [ ] Реализовать `tinvest.Adapter`, удовлетворяющий `broker.Broker` (gRPC).
- [ ] Реализовать `tinvest.MarketDataAdapter`, удовлетворяющий `broker.MarketData` (gRPC streaming или WebSocket).
- [ ] Авторизация: bearer API-токен.
- [ ] Регистрация в `BrokerRegistry`.

### 5.4 Тесты

- [ ] Unit-тесты для логики diff конфигов.
- [ ] Интеграционный тест: изменение файла конфига, проверка корректной перезагрузки оркестратора.
- [ ] Интеграционные тесты адаптера Т-Инвестиций (на sandbox API, если доступен).

**Результат Фазы 5:** Изменения конфига и стратегий подхватываются автоматически. Т-Инвестиции доступны как второй брокер.

## Фаза 6: Пул subprocess-процессов и подготовка к масштабу

**Цель:** Оптимизация использования ресурсов для 1000+ ботов.

**Оценка трудозатрат:** 2 недели

### 6.1 Пул subprocess-процессов

- [ ] Реализовать `subprocess.Pool`: управляет фиксированным пулом worker-процессов.
- [ ] Cron-боты заимствуют worker на время выполнения, возвращают после.
- [ ] Active-боты получают выделенные долгоживущие процессы (минуя пул).
- [ ] Пулы per-language (пул Python, пул Node и т.д.).
- [ ] Настраиваемые размеры пулов.

### 6.2 Оптимизация памяти

- [ ] `sync.Pool` для объектов `State` и `Intent`.
- [ ] Профилирование и тюнинг `GOGC` / `GOMEMLIMIT` для целевого количества ботов.
- [ ] Бенчмарк с 100, 500, 1000 ботами (микс wasm/subprocess/cron/active).

### 6.3 Наблюдаемость

- [ ] Метрики Prometheus: активные боты, обработанные тики/сек, исполненные intent'ы, ошибки.
- [ ] Health check эндпоинт (HTTP) для мониторинга.
- [ ] Отчёт о статусе per-bot (работает, ошибка, время последнего выполнения).

### 6.4 Тесты

- [ ] Нагрузочный тест: 1000 ботов с мок-брокером, измерение памяти, CPU, пауз GC.
- [ ] Стресс-тест: быстрые изменения конфига при работающих ботах.

**Результат Фазы 6:** Система доказанно выдерживает 1000+ ботов с мониторингом и пулингом процессов.

## Технический долг (закрывать инкрементально)

Эти задачи следует решать постепенно, не выделяя в отдельную фазу:

- [ ] **Тесты для адаптера БКС** — сейчас нулевое покрытие тестами. Добавить unit-тесты с HTTP-мокированием.
- [ ] **Учёт лотности** — текущая логика ребалансировки не учитывает размер лота инструментов. Исправить в валидации intent'ов `BrokerProxy`.
- [ ] **Retry/backoff** — добавить экспоненциальный backoff для транзиентных ошибок брокера (`ErrTimeout`, `ErrRateLimited`, `ErrUnavailable`).
